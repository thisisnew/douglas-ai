# Process

## 1단계. 문서 분석

입력 문서 유형에 따라 다음 정보를 추출한다.

| 입력 유형 | 분석 방법 | 추출 항목 |
|---|---|---|
| Figma URL | Figma 기반 분석 | Component, Text, Input 속성, Interactive 흐름 |
| Axure URL | 웹 흐름 분석 | Widget Notes, Interaction, Dynamic Panel |
| Jira/Confluence | 요구사항 문서 분석 | 설명, Acceptance Criteria, 첨부파일, 연결 이슈 |
| 일반 URL | 웹 화면 분석 | UI 구성요소, 라벨, 안내문구, 화면 전환 경로 |
| 스크린샷/PPT/PDF | 화면/문서 분석 | UI 구성요소, 사용자 플로우, 주요 액션, 상태 |

이 단계에서 반드시 확인할 항목:

- 화면명
- 기능 목적
- 주요 사용자 역할
- 입력 필드
- 버튼 및 액션
- 상태값 변화
- 오류 메시지
- 출력/다운로드/발송 여부
- 인증/권한 필요 여부

## 2단계. Section 구조 설계

Section은 최대 3 depth까지 사용한다.  
과도한 중첩은 금지한다.

예시:

좋은 예:
- 로그인
  - 정상 케이스
  - 오류 케이스
  - 엣지 케이스

나쁜 예:
- 회원관리 > 인증 > 로그인 > 입력검증 > 이메일 > 특수문자

Section 설계 원칙:

1. 기능 단위로 최상위 Section을 만든다.
2. 그 아래에 정상/오류/엣지/보안 등으로 나눈다.
3. 너무 세밀한 기술적 분류는 피한다.

## 3단계. 테스트 케이스 작성

### UI/UX 테스트케이스 작성
다음을 포함한다.

- 정상 케이스
- 오류 케이스
- 엣지 케이스
- 상태 변화
- 사용자 흐름
- 버튼/링크/모달 동작
- 다운로드/출력/발송 기능
- 일괄 처리 기능
- 데이터 표시/정렬/필터/검색

### 보안 테스트케이스 작성
다음을 포함한다.

- 입력값 공격
- 인증/인가 우회
- 데이터 접근 통제
- URL 직접 접근
- 파라미터 조작
- 조직 간 접근 제한

## 4단계. 보조 문서 작성

### 질문 문서 작성
아래에 해당하면 반드시 질문 문서에 기록한다.

- 정책이 문서에 없음
- 권한 조건이 불명확함
- 성공/실패 메시지가 없음
- 입력 제한 조건이 없음
- 날짜 정책이 없음
- 예외 동작이 정의되지 않음

### 리스크 분석 문서 작성
기능 실패 시 영향이 큰 항목을 정리한다.

예:
- 데이터 유실
- 잘못된 승인 처리
- 타 사용자 데이터 노출
- 다운로드 파일 오표기
- 상태값 불일치

### 체크리스트 작성
최종적으로 누락 여부를 확인할 수 있는 체크리스트를 정리한다.

## 5단계. 검증 및 출력

최종 출력 전 아래 항목을 점검한다.

- CSV 포맷이 TestRail 업로드 형식과 맞는가
- UI/UX와 보안 파일이 분리되어 있는가
- 필수 보안 테스트가 포함되어 있는가
- Preconditions가 충분히 구체적인가
- Expected Result가 측정 가능한가
- 질문이 필요한 항목이 질문 문서에 정리되어 있는가
