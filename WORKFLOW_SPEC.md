# DOUGLAS 공식 워크플로우 명세서

## 1. 목적

DOUGLAS는 사용자의 입력을 해석하여 적절한 작업 모드로 분류하고, 필요한 경우 적합한 에이전트를 매칭하여 질의응답, 토론, 구현, 조사, 문서 생성 등의 작업을 수행하는 멀티에이전트 기반 워크플로우 시스템이다.

본 명세서의 목적은 다음과 같다.

- 사용자 입력을 일관된 기준으로 처리한다.
- 방 생성, 에이전트 매칭, 승인, 실행, 후속처리의 전체 흐름을 표준화한다.
- 단순 성공 시나리오뿐 아니라 불명확한 요청, 실패, 사용자 개입, 중단, 롤백 등의 예외 흐름까지 정의한다.
- UX와 내부 상태 전이를 분리하여 확장 가능하고 구현 가능한 상태 머신 기반 구조를 제공한다.

## 2. 핵심 원칙

### 2.1 사용자 중심 원칙

- 사용자는 언제든지 작업 도중 의견을 추가할 수 있다.
- 사용자는 필요한 경우 에이전트를 직접 선택할 수 있다.
- 사용자는 되돌릴 수 없는 단계 이전에 반드시 승인할 수 있어야 한다.
- 사용자는 언제든지 작업을 중단하거나 이전 단계로 되돌리도록 요청할 수 있다.

### 2.2 실행 안정성 원칙

- 모든 요청은 먼저 의도(intent)를 판별한다.
- 의도가 불명확하면 실행보다 재질문이 우선한다.
- 비용이 큰 작업은 반드시 계획 수립과 승인 절차를 거친다.
- 되돌릴 수 없는 작업은 별도 승인 없이는 수행하지 않는다.

### 2.3 멀티에이전트 원칙

- 에이전트는 역할에 따라 매칭한다.
- 단일 에이전트로 충분한 경우 불필요한 멀티에이전트 구성을 피한다.
- 복수 에이전트는 관점 충돌, 전문영역 분리, 검토 필요성이 있을 때만 사용한다.
- 문서 결과물의 최종 정리는 문서 생성 전문가 또는 그에 준하는 에이전트가 담당한다.

### 2.4 상태 기반 원칙

- DOUGLAS는 케이스 나열형이 아니라 상태 머신 기반으로 동작한다.
- 각 요청은 현재 상태, 승인 상태, 실행 권한 상태를 가진다.
- 후속 요청은 항상 기존 상태와 맥락 위에서 해석된다.

## 3. 주요 개념 정의

### 3.1 요청(Request)

사용자가 입력한 프롬프트, 파일, URL, 이미지, 후속 메시지 등을 포함하는 작업 단위.

### 3.2 방(Room)

특정 요청을 중심으로 생성되는 대화/작업 단위 공간.
DOUGLAS와 에이전트, 사용자 간 상호작용은 기본적으로 방 단위로 관리한다.

### 3.3 에이전트(Agent)

특정 역할, 전문성, 모델, 업무규칙을 가진 작업 주체.

예시:
- 백엔드 전문가
- 프론트엔드 전문가
- QA 전문가
- 문서 생성 전문가
- 조사 전문가
- 기획 전문가

### 3.4 작업 모드(Task Mode)

요청의 본질적인 성격을 나타내는 분류.

종류:
- 질의응답
- 토론
- 구현
- 조사
- 문서화
- 복합 요청

### 3.5 출력 모드(Output Mode)

작업 결과가 어떤 형태로 제공되는지 나타내는 분류.

종류:
- 채팅 응답
- 방 내 대화
- 코드/구현 결과
- 문서 파일
- 요약본
- 후속 작업 전환

### 3.6 승인(Approval)

사용자의 명시적 동의가 필요한 게이트.

종류:
- 에이전트 승인
- 계획 승인
- 실행 권한 승인
- 최종 승인
- 되돌릴 수 없는 단계 직전 승인

## 4. Intent 분류 체계

### 4.1 기본 분류

모든 요청은 아래 작업 모드 중 하나 이상으로 분류한다.

| 작업 모드 | 설명 |
|-----------|------|
| 질의응답 | 단일 질문에 대해 답하거나 설명하는 요청 |
| 토론 | 여러 관점 검토, 아이디어 비교, 대안 도출, 설계 논의를 포함하는 요청 |
| 구현 | 실제 산출물 생성, 코드 작성, 수정, 설계 반영, 작업 수행이 필요한 요청 |
| 조사 | 자료 수집, 검색, 비교, 정리 등이 중심인 요청 |
| 문서화 | 문서 파일 작성, 회의록/기획서/설계서/제안서 등의 결과물 생성 요청 |
| 복합 요청 | 둘 이상의 작업 모드가 섞여 있는 요청 (예: "문제점 토론해보고 개선안을 문서로 정리해줘") |

### 4.2 분류 신뢰도

Intent 분류 결과에는 반드시 신뢰도를 함께 가진다.

- **HIGH**: 거의 확실함
- **MEDIUM**: 유력하나 다른 가능성 존재
- **LOW**: 불명확하거나 중의적임

### 4.3 불명확 처리 규칙

아래 중 하나라도 해당하면 불명확 요청으로 간주한다.

- URL만 입력된 경우
- 이미지만 업로드되고 목적이 없는 경우
- 파일만 올리고 지시가 없는 경우
- "이거 해줘", "이거 봐줘"처럼 행위 목적이 모호한 경우
- 질의응답/토론/구현/문서화 중 어느 것인지 판별이 어려운 경우

불명확 요청일 경우:
1. DOUGLAS가 재질문한다.
2. 재질문 후에도 모호하면 가능한 선택지를 제시한다.
3. 사용자의 응답이 오기 전까지 실행 단계로 넘어가지 않는다.

## 5. 방(Room) 정책

### 5.1 방 생성 원칙

방 생성은 별도 케이스가 아니라 공통 lifecycle이다.

기본 규칙:
- 질의응답, 토론, 구현, 조사, 문서화 대부분은 방 단위로 처리한다.
- 구현, 토론, 조사, 문서화는 원칙적으로 방 생성이 필요하다.
- 아주 단순한 즉답성 질의응답은 예외적으로 방 없이 처리할 수 있으나, 후속 확장 가능성을 고려해 기본은 방 생성이다.

### 5.2 방 생성 시점

Intent가 판별되고, 요청을 독립 작업 단위로 처리할 필요가 있을 때 방을 생성한다.

### 5.3 방의 역할

방은 다음을 보존한다.
- 사용자 원 요청
- 분류된 intent
- 매칭된 에이전트 목록
- 토론 기록
- 실행 계획
- 승인 이력
- 작업 로그
- 후속 요청 맥락
- 생성된 산출물 참조

## 6. 에이전트 매칭 정책

### 6.1 기본 원칙

DOUGLAS는 요청에 따라 가장 적합한 에이전트를 자동 매칭한다.

### 6.2 단일 에이전트 선택 기준

아래 조건이면 단일 에이전트를 우선한다.
- 전문영역이 명확함
- 여러 관점 충돌이 필요하지 않음
- 작업 범위가 작음
- 질의응답 또는 단순 구현임

### 6.3 복수 에이전트 선택 기준

아래 조건이면 복수 에이전트를 고려한다.
- 복수 전문 영역이 동시에 필요함
- 상충되는 관점 검토가 필요함
- 검토/비판/보완 라운드가 유의미함
- 구현 전에 설계 토론이 중요함

예:
- 백엔드 + 프론트엔드
- 개발 + QA
- 기획 + 개발
- 조사 + 문서화

### 6.4 에이전트 승인 정책

모든 경우에 사용자 확인을 강제하지는 않는다.

**자동 진행 조건** — 아래 조건을 모두 만족하면 자동 진행 가능:
- intent 신뢰도 HIGH
- agent match 신뢰도 HIGH
- 저위험 작업
- 비용이 낮은 작업

**사용자 확인 조건** — 아래 중 하나라도 해당하면 사용자 확인을 요청한다:
- intent 신뢰도 MEDIUM 또는 LOW
- agent match 후보가 2개 이상 경합함
- 구현 작업임
- 복수 에이전트 토론임
- 파일 수정/문서 생성/외부도구 사용이 포함됨

### 6.5 매칭 실패 시 처리

- 적절한 에이전트를 찾지 못하면 사용자에게 후보를 제시한다.
- 후보가 없으면 직접 선택하도록 유도한다.
- 필요 시 새 에이전트 생성 또는 등록을 제안한다.

## 7. 공통 상태 정의

각 요청은 아래 상태 중 하나를 가진다.

| 상태 | 설명 |
|------|------|
| RECEIVED | 입력 수신 |
| INTENT_CLASSIFIED | 의도 분류 완료 |
| WAITING_CLARIFICATION | 불명확하여 재질문 대기 |
| ROOM_CREATED | 방 생성 완료 |
| AGENT_MATCHED | 에이전트 매칭 완료 |
| WAITING_AGENT_CONFIRMATION | 에이전트 확인 대기 |
| DISCUSSING | 토론 진행 중 |
| PLANNING | 계획 수립 중 |
| WAITING_PLAN_APPROVAL | 계획 승인 대기 |
| WAITING_EXECUTION_APPROVAL | 실행 권한 승인 대기 |
| EXECUTING | 실행 중 |
| WAITING_USER_FEEDBACK | 사용자 피드백 대기 |
| WAITING_FINAL_APPROVAL | 최종 승인 대기 |
| DOCUMENTING | 문서 생성 중 |
| COMPLETED | 완료 |
| FAILED | 실패 |
| CANCELLED | 취소 |

## 8. 승인 및 권한 모델

### 8.1 승인 종류

| 승인 유형 | 설명 |
|-----------|------|
| 에이전트 승인 | 매칭된 에이전트 구성으로 진행할지 사용자에게 묻는 승인 |
| 계획 승인 | 실행 전에 도출된 계획, 범위, 순서, 산출물에 대해 받는 승인 |
| 실행 권한 승인 | 실제 파일 수정, 문서 생성, 도구 실행 등 행위를 허용하는 승인 |
| 최종 승인 | 마무리 직전 결과 방향이 괜찮은지 확인하는 승인 |
| 되돌릴 수 없는 단계 직전 승인 | 최종 파일 저장, 문서 export, 대규모 반영, 외부 전송, 커밋/배포/PR 생성 등 확정 행위 전 필수 승인 |

### 8.2 실행 권한 상태

| 권한 | 설명 |
|------|------|
| READ_ONLY | 읽기 전용 |
| PLAN_ONLY | 계획만 가능 |
| PATCH_ALLOWED | 패치 허용 |
| FILE_WRITE_ALLOWED | 파일 쓰기 허용 |
| TOOL_ALLOWED | 도구 사용 허용 |
| FULL_EXECUTION_ALLOWED | 전체 실행 허용 |

## 9. 후속 입력 처리 규칙

작업 도중 사용자가 추가로 입력한 메시지는 아래 4종으로 분류한다.

### 9.1 즉시 반영 가능
현재 단계에서 바로 반영 가능한 미세 조정.
예: 문구 변경, 변수명 변경, 형식 수정, 응답 톤 변경

### 9.2 다음 단계 반영
현재 단계는 유지하고 이후 단계부터 반영.
예: 이후부터 테스트를 우선해달라, 다음 단계에서 문서도 같이 만들어달라

### 9.3 계획 재수립 필요
기존 계획을 다시 짜야 하는 수준의 변경.
예: 기술 스택 변경, 구현 범위 확대, 역할 구조 변경, 토론에서 구현으로 전환

### 9.4 롤백 필요
이미 수행된 결과를 되돌려야 하는 요청.
예: 방금 수정 취소, 이전 단계로 복귀, 마지막 반영 무시

후속 입력이 들어오면 DOUGLAS는 이를 위 4가지 중 하나로 분류하여 처리한다.

## 10. 토론 종료 규칙

복수 에이전트 토론은 무한히 지속되지 않는다.
아래 조건 중 하나를 만족하면 종료한다.

- 명시적 합의 도달
- 결론 후보들이 충분히 정리됨
- 최대 라운드 수 도달
- 사용자 개입 요청 필요
- 더 이상 새로운 근거가 나오지 않음

토론 종료 시 DOUGLAS는 반드시 아래 형태로 정리한다.
- 결론
- 대안
- 미해결 쟁점
- 추천안
- 다음 액션

## 11. 완료 기준 정의

"완료"는 작업마다 다르므로 반드시 명시적으로 판단한다.

### 11.1 질의응답 완료
- 질문에 대한 직접 응답 제공
- 필요한 설명 또는 근거 포함
- 사용자가 후속 질문 가능한 상태

### 11.2 토론 완료
- 결론 또는 선택지 정리 완료
- 핵심 쟁점 정리 완료
- 사용자 의사결정에 필요한 수준 확보

### 11.3 구현 완료
아래 중 요청에 맞는 기준을 충족해야 완료로 본다.
- 초안 작성 완료
- 코드 수정 완료
- 구조 제안 완료
- 테스트 가능 상태
- 최종 반영 직전 상태
- 사용자 승인 후 최종 반영 완료

### 11.4 문서 생성 완료
- 문서 파일 생성 완료
- 저장 경로 확보
- 사용자에게 완료 및 경로 안내

## 12. 워크플로우 상세

### 12.1 공통 워크플로우

1. 사용자 입력 수신
2. intent 분류
3. 불명확 시 재질문
4. 방 생성 여부 결정 및 방 생성
5. 에이전트 매칭
6. 필요 시 사용자에게 에이전트 확인
7. 작업 모드에 따라 질의응답/토론/구현/조사/문서화 진행
8. 필요 시 계획 수립 및 승인
9. 실행 진행
10. 필요 시 최종 승인 또는 되돌릴 수 없는 단계 직전 승인
11. 결과 정리
12. 후속처리 가능 상태로 전환

### 12.2 질의응답 워크플로우

**적용 조건**: 사용자의 요청이 설명, 답변, 해석, 비교, 간단한 조언 중심일 때

**흐름**:
1. 프롬프트 입력
2. 질의응답 intent 판별
3. 방 생성
4. 에이전트 자동 매칭 또는 DOUGLAS 직접 응답 여부 결정
5. 필요 시 사용자에게 에이전트 확인
6. 에이전트가 답변 수행
7. 결과 응답
8. 후속처리 가능 상태로 전환

**비고**:
- 단일 에이전트 토론과 경계가 애매한 경우, 질의응답으로 간주하되 심화 질문을 허용한다.
- 비용이 작고 신뢰도가 높으면 에이전트 확인 없이 바로 진행 가능하다.

### 12.3 토론 워크플로우

**적용 조건**: 여러 관점 비교, 설계 검토, 아이디어 브레인스토밍, 장단점 분석, 선택지 도출

**흐름**:
1. 프롬프트 입력
2. 토론 intent 판별
3. 방 생성
4. 에이전트 매칭
5. 필요 시 사용자에게 에이전트 확인
6. 단일 에이전트면 심화 질의응답처럼 진행
7. 복수 에이전트면 라운드 기반 토론 시작
8. 각 라운드마다 사용자 개입 가능
9. 종료 조건 충족 시 토론 종료
10. DOUGLAS가 결론, 대안, 미해결 쟁점 정리
11. 결과 응답
12. 후속처리 가능 상태로 전환

**토론 라운드 규칙**:
- 각 라운드는 명확한 주제를 가진다.
- 같은 주장 반복만 발생하면 종료를 고려한다.
- 대립이 지속되면 사용자 선택을 요청한다.

### 12.4 구현 워크플로우

**적용 조건**: 실제 산출물 생성, 코드 수정 또는 작성, 구조 설계 반영, 실질적 작업 수행

**흐름**:
1. 프롬프트 입력
2. 구현 intent 판별
3. 방 생성
4. 에이전트 매칭
5. 필요 시 사용자에게 에이전트 확인
6. 단일 또는 복수 에이전트 구성
7. 복수 에이전트일 경우 사전 토론 가능
8. DOUGLAS가 실행 계획 도출
9. 사용자에게 계획 승인 요청
10. 필요 시 실행 권한 승인 요청
11. 승인되면 작업 수행
12. 진행 중 사용자는 추가 요구사항 입력 가능
13. 추가 요구사항은 즉시반영/다음단계반영/계획재수립/롤백 중 하나로 처리
14. 마지막 단계 또는 되돌릴 수 없는 단계 직전 승인 요청
15. 승인되면 마무리 수행
16. 결과 정리
17. 후속처리 가능 상태로 전환

**구현 필수 규칙**:
- 계획 승인 없이 고비용 구현을 시작하지 않는다.
- 되돌릴 수 없는 작업은 직전 승인 없이 수행하지 않는다.
- 진행 중 범위 변경이 크면 반드시 계획 재수립 상태로 되돌아간다.
- 사용자가 원하면 특정 단계로 복귀할 수 있다.

**필수 UX 문구 원칙**:
마지막 단계 직전에는 반드시 다음 취지의 확인을 수행한다.
예: "이제 마지막 단계입니다. 여기까지의 방향이 괜찮다면 승인해주시면 마무리하겠습니다."

### 12.5 조사 워크플로우

**적용 조건**: 정보 수집, 비교 정리, 사례 조사, 레퍼런스 탐색

**흐름**:
1. 프롬프트 입력
2. 조사 intent 판별
3. 방 생성
4. 조사에 적합한 에이전트 매칭
5. 필요 시 사용자에게 범위 확인
6. 조사 수행
7. 결과 요약 및 정리
8. 필요 시 문서화 또는 토론/구현으로 전환
9. 후속처리 가능 상태로 전환

### 12.6 문서 생성 워크플로우

**적용 조건**: 문서 파일 생성 요청, 기존 대화를 기반으로 문서화, 조사 결과를 문서로 정리

**문서 생성 원칙**:
- 최종 문서화와 export는 문서 생성 전문가가 담당한다.
- 내용 초안은 해당 도메인 에이전트가 만들 수 있다.
- 관련 없는 에이전트가 임의로 최종 문서 생성을 담당해서는 안 된다.

**흐름 A: 작업 후 문서 생성**
1. 기존 방에서 "문서로 만들어줘" 요청 발생
2. DOUGLAS가 해당 방의 대화, 결론, 산출물 요약
3. 문서 생성 전문가가 문서 구조화 및 작성
4. 문서 생성 완료
5. 사용자에게 간단히 완료 메시지와 경로 안내

**흐름 B: 처음부터 문서 생성 요청**
1. 프롬프트 입력
2. 문서화 또는 조사+문서화 intent 판별
3. 방 생성
4. 필요 시 조사 수행
5. 문서 초안 생성
6. 문서 생성 전문가가 정리 및 최종화
7. 문서 생성 완료
8. 사용자에게 간단히 완료 메시지와 경로 안내

**출력 규칙**:
- 문서 생성 완료 시 방 안에 문서 본문 전체를 불필요하게 다시 장문으로 보여줄 필요는 없다.
- 기본 응답 형식: "완료되었습니다." + "저장 경로: [경로]"
- 단, 사용자가 본문 확인을 요청하면 예외적으로 본문 미리보기를 제공할 수 있다.

## 13. 후속처리 워크플로우

모든 작업이 완료된 이후에도 후속처리를 지원한다.

후속 요청은 아래 중 하나로 전환될 수 있다.
- 추가 질의응답
- 토론으로 전환
- 구현으로 전환
- 조사로 전환
- 문서 생성 요청
- 기존 결과 수정 요청
- 다른 에이전트 재투입 요청

후속 요청은 반드시 기존 방의 맥락을 이어받아 처리한다.
단, 사용자가 새 방을 원하면 별도 방으로 분리할 수 있다.

## 14. 예외 처리 명세

### 14.1 요건 불명확
- DOUGLAS가 재질문한다.
- 목적, 원하는 결과물, 작업 방식 중 빠진 정보를 보완받는다.
- 재질문 전까지 실행하지 않는다.

### 14.2 에이전트 매칭 실패
- 후보 에이전트를 제시한다.
- 사용자 직접 선택을 요청한다.
- 필요 시 새 에이전트 생성 제안한다.

### 14.3 토론 무한 루프
- 동일 주장 반복, 실질적 진전 없음, 최대 라운드 초과 시 종료한다.
- DOUGLAS가 쟁점만 요약하고 사용자 선택을 요청한다.

### 14.4 구현 불가
아래 사유가 있으면 구현 불가 상태로 전환한다.
- 정보 부족
- 권한 부족
- 파일 접근 불가
- 도구 부족
- 요구사항 충돌
- 외부 환경 문제

이 경우 DOUGLAS는 반드시 아래를 알려야 한다.
- 어디까지 수행했는지
- 무엇이 막혔는지
- 다음에 무엇이 필요한지

### 14.5 사용자 응답 없음
- 승인 대기 상태 유지
- 현재 계획과 진행 상태를 보존
- 사용자가 다시 입력하면 이어서 재개

### 14.6 사용자 중간 취소
- 즉시 작업 중단
- 현재 결과물 보존 여부를 결정
- 상태를 CANCELLED로 전환

### 14.7 롤백 요청
- 가능한 롤백 범위를 안내
- 특정 단계로 복귀
- 계획 재수립 또는 실행 재개

## 15. 사용자 경험(UX) 문구 원칙

### 15.1 에이전트 확인
예:
- "이 요청은 백엔드 전문가가 가장 적합해 보입니다. 이 에이전트로 진행할까요?"
- "프론트엔드와 QA 두 에이전트가 함께 보는 것이 적절해 보입니다. 이 조합으로 진행할까요?"

### 15.2 계획 승인
- 계획 승인 대기 시 PlanCard가 편집 모드로 전환됨.
- 사용자가 단계 텍스트를 직접 편집(tap → inline TextField), 삭제, 추가, 순서 변경 가능.
- 편집된 계획은 LLM 재생성 없이 그대로 승인하여 실행 가능.
- "수정 요청" 클릭 시 피드백과 함께 LLM에 재계획 요청.
- ApprovalCard는 compact summary만 표시 (PlanCard와 중복 방지).
예:
- "아래 순서로 진행하려고 합니다. 괜찮으면 승인해주세요."
- "범위와 단계는 다음과 같습니다. 수정할 부분이 있으면 말씀해주세요."

### 15.3 실행 중 사용자 개입 (실시간 피드백 루프)
- 실행 중에도 입력 영역이 표시되어 사용자가 실시간으로 추가 요건을 입력할 수 있음.
- 각 단계 완료 후 사용자 메시지를 확인하여, 새 요건이 있으면 같은 단계를 재실행.
- "3단계부터 다시" 같은 롤백 요청도 실행 중 입력으로 처리 가능.
- PlanCard에서 완료된 단계를 클릭하여 해당 단계로 롤백 가능.
예:
- "추가 요건을 반영하여 단계 3을 다시 실행합니다."
- "단계 2부터 다시 실행합니다."

### 15.4 마지막 단계 승인
- 실시간 피드백 루프가 마지막 단계 승인 게이트를 대체.
- 사용자는 매 단계에서 개입 기회가 있으므로 별도 승인 없이 자연스럽게 완료.
- 완료 후 불만이 있으면 입력 영역에서 후속 요청 가능 (follow-up cycle).

### 15.5 문서 생성 완료
예:
- "완료되었습니다."
- "저장 경로: {path}"

## 16. 내부 판단 규칙

### 16.1 질의응답과 토론의 경계
- 단일 질문에 대한 응답이면 질의응답
- 비교, 대안, 관점 충돌, 논의가 핵심이면 토론

### 16.2 질의응답과 구현의 경계
- 설명, 예시, 조언 수준이면 질의응답
- 실제 결과물 작성, 수정, 생산이 필요하면 구현

### 16.3 구현과 문서화의 경계
- 실질 산출물이 코드/설계 반영이면 구현
- 최종 결과물이 문서 파일이면 문서화
- 둘 다 필요하면 복합 요청으로 처리

### 16.4 단일/복수 에이전트 판단
- 하나의 전문성으로 충분하면 단일
- 검토, 협업, 상충 관점이 필요하면 복수

## 18. 상태 전이 요약

### 기본 흐름
```
RECEIVED
→ INTENT_CLASSIFIED
→ ROOM_CREATED
→ AGENT_MATCHED
→ 필요 시 WAITING_AGENT_CONFIRMATION
→ 작업에 따라 DISCUSSING 또는 PLANNING
→ WAITING_PLAN_APPROVAL
→ 필요 시 WAITING_EXECUTION_APPROVAL
→ EXECUTING
→ 필요 시 WAITING_USER_FEEDBACK
→ 필요 시 WAITING_FINAL_APPROVAL
→ 필요 시 DOCUMENTING
→ COMPLETED
```

### 예외 흐름
- 불명확: WAITING_CLARIFICATION
- 실패: FAILED
- 취소: CANCELLED

## 19. 공식 케이스 요약

### 케이스 A: 질의응답
질문 중심 요청. 단일 에이전트 또는 DOUGLAS가 응답. 필요 시 에이전트 확인 후 진행.

### 케이스 B: 단일 에이전트 토론
실질적으로 심화 질의응답에 가까운 토론. 단일 전문가가 논리적으로 답변과 검토를 수행.

### 케이스 C: 복수 에이전트 토론
복수 전문가가 라운드 기반으로 토론. 사용자는 각 라운드마다 개입 가능. 종료 시 결론과 대안을 정리.

### 케이스 D: 단일 에이전트 구현
계획 수립 → 승인 → 작업 → 최종 승인 → 완료

### 케이스 E: 복수 에이전트 구현
사전 토론 가능 → 계획 수립 → 승인 → 작업 → 최종 승인 → 완료

### 케이스 F: 문서 생성
기존 방의 내용을 문서화하거나, 처음부터 조사 후 문서 생성. 최종 문서화는 문서 생성 전문가가 담당.

### 케이스 G: 후속처리
완료 후에도 같은 방에서 질의응답, 토론, 구현, 문서화로 확장 가능.

### 케이스 H: 요건 불명확
재질문 후 명확해질 때까지 실행 보류.

## 20. 최종 원칙

1. DOUGLAS는 먼저 분류하고, 그 다음 매칭하고, 그 다음 실행한다.
2. 불명확한 상태에서 성급히 실행하지 않는다.
3. 고비용 작업은 반드시 계획 승인 단계를 거친다.
4. 되돌릴 수 없는 작업은 반드시 직전 승인 단계를 거친다.
5. 사용자 후속 입력은 항상 기존 맥락 위에서 해석한다.
6. 토론은 무한 반복하지 않고 종료 조건을 가진다.
7. 구현은 완료 기준이 명확해야 한다.
8. 문서 생성은 최종적으로 문서 전문가가 책임진다.
9. 예외 흐름은 정상 흐름만큼 중요하게 다룬다.
10. 전체 시스템은 케이스 나열보다 상태 머신으로 구현한다.
